# Telegram Integration

## This Integration Is Optional And Starts Disabled
Telegram is modeled as an optional outbound operator channel. The integration surface exists in config and matrix policy before any sender logic is enabled.

## This Integration Uses Config Keys Rather Than Ad Hoc Secrets
`runtime/config/project-vela.yaml` now includes:

- `integrations.telegram`
- `telegram.bot_token`
- `telegram.default_chat_id`
- `telegram.send_morning_report`
- `telegram.send_blocked_summary`

These fields define whether Telegram is enabled, where messages should go, and which report classes are allowed to leave the matrix.

## This Integration Is Intended For Governed Outbound Messaging
Telegram is the likely home for:

- DC morning reports
- blocked-item summaries that need Dario
- optional patrol or night-cycle alerting

The message path should stay governed. Telegram should not become a side channel that bypasses approval, validation, or matrix logging.

## This Integration Surface Exists Before Sender Logic
The repo now has a canonical place for Telegram configuration and policy, but not yet a bot sender implementation. Actual send operations still need:

- a governed sender adapter
- service or CLI send surfaces
- explicit tests for missing config, allowed sends, and refusal behavior
