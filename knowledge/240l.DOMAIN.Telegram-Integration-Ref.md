# Telegram Integration

## This Integration Is Optional And Starts Disabled
Telegram is modeled as an optional outbound operator channel. It stays disabled until the local config explicitly enables it.

## This Integration Uses Config Keys Rather Than Ad Hoc Secrets
`runtime/config/project-vela.yaml` now includes:

- `integrations.telegram`
- `telegram.bot_token`
- `telegram.default_chat_id`
- `telegram.send_morning_report`
- `telegram.send_blocked_summary`

These fields define whether Telegram is enabled, where messages should go, and which report classes are allowed to leave the matrix.

## This Integration Is Intended For Governed Outbound Messaging
Telegram is the likely home for:

- DC morning reports
- blocked-item summaries that need Dario
- optional patrol or night-cycle alerting

The message path should stay governed. Telegram should not become a side channel that bypasses approval, validation, or matrix logging.

## This Integration Now Supports Governed Test And Night Cycle Delivery
The current implementation supports:

- governed manual test sends
- optional DC night report delivery
- optional blocked-summary delivery

The message path still remains outside the matrix for delivery only. The canonical record remains the report or event already written inside the matrix.

## This Integration Still Needs Broader Delivery Surfaces Over Time
Telegram now has a sender path, but it may still grow into:

- explicit morning-brief commands
- patrol alert delivery
- richer routing by chat or topic
