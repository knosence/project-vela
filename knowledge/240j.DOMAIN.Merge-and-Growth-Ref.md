# Merge And Growth Protocol

## This Protocol Decides When Content Should Merge And When Structure Should Grow
The matrix prefers one canonical home, then controlled growth. Growth is earned. Fragmentation is not.

## This Protocol Merges Shared Subjects Once They Appear Across Three Or More Entities
If the same subject appears as refs or equivalent pointer-like representations in three or more entities, that subject should be merged into one canonical SoT.

## This Protocol Checks Merge Before Spawn
Before creating a new SoT, search the matrix for an existing subject home. If one exists, merge into it. If no home exists, or if the current home has reached the next growth threshold, then grow.

## This Protocol Climbs The Growth Ladder One Step At A Time
The growth path is flat, then fractal, then reference, then spawned SoT. Use the lightest intervention that restores clarity.

## This Protocol Uses The Next Available Local ID Rather Than Imported IDs
Numbering is local to this matrix. New SoTs and refs take the next available local ID for their branch. IDs from another matrix are not imported.

## This Protocol Allows ID Reuse Only After The Original File Truly No Longer Exists
If an ID is still present anywhere in the matrix, it is occupied. If the file no longer exists, that slot may be reused.
